1. Abstract

This webpage is written in accordance with the Scholarly HTML template - a domain-specific rich document format. The role of this documentation is to present the esential functionalities of our website and its user interaction. The structure follows the IEEE System Requirements Specification Template.

For detailed information on these templates, please refer to the Scholarly HTML and IEEE System Requirements Specification Template.

It's important to note that this document offers some screenshots that ore only displayed as a proof of concept. The final design should be checked in the product itself.

2. Introduction

2.1 Purpose

This Software Requirements Specification (SRS) document outlines the specifications for the development of a web application aggregator intitulated M-PIC. The purpose of this document is to clearly define the scope, objectives, and goals of the project, including the management of photographs stored across various platforms that allow photo or video distribution, such as Instagram and Youtube.

2.2 Document conventions

The SRS follows standard document conventions, including clear formatting, headings, and section numbering for easy navigation. In this document, the typographical conventions follow the Roboto font, with the fallback font of sans-serif.

2.3 Intended Audience and Reading Suggestions

This document is intended for the two developers of the site involved in the development and deployment of the web application, serving as an outline for the overall website. The targeted audience also includes the users of the website and the coordonator(s) of the project (Web technologies professors from UAIC University). Readers are encouraged to start with the Introduction chapter and progress to sections relevant to their role or interest.

2.4 Product Shape

The software being specified is a web application aggregator designed to facilitate the management of photographs across multiple platforms. Its purpose is to allow users to retrieve personal images from the two mentioned platforms, along with associated comments, ratings, and other relevant metadata such as tags, share counts, and dimensions. If necessary, additional supported platforms will be added. The application will also provide advanced search capabilities and enable users to upload new photographs to these platforms. Additionally, users will have the ability to edit graphical content within the application using CSS filters and masks.

3. Overall Description

3.1 Product Perspective

This product is designed to be used as an assessment method for the Web Technologies subject. It is developed from scratch, without using any pre-made frameworks from the internet. However, it will use its own primitive framework in order to simplify the product deployment. Therefore, this website should be self-contained in the front-end part. As for the back-end, this website will use some external APIs to allow the average end-user to connect to different social media platforms, in order to aggregate its files.

3.2 Product Functions

Our main product functions are:

  • Allow an user to log in/sign up to his/her account.
  • Ability to aggregate and manage the photos from different social media platforms.
  • View the number of likes and comments from each photo.
  • View other relevant information, such as the photo dimensions, post date, etc.
  • Multi-criteria search capabilities.
  • Apply filters or masks to the aggregated photos.
  • Ability to select a number of photos to create a photo collage.
  • Allow the user to upload videos or photos to our supported applications.

In the picture from above is presented a rudimentary user flow diagram. It shows the main capabilities of the website, as mentioned before in the list. If the photo is too small, it can also be visualized here in a new tab.

3.3 Operating Environment

In the first part of the development, the website will be stored locally on the users' personal computer. It can be downloaded from this GitHub page.

As for the end user perspective, this product shouldn't require too high specifications, as an internet connection and a web browser will suffice. Even if the used browser isn't compatible with all of the product's features, there should be some fallback methods, allowind the user experience to be as close as possible to the maximum potentional of the website.

It's important to note that our product will be available on any type of device (PCs, phones, tablets, etc.). Therefore, its design shall be responsive.

3.4 Design and Implementation Constraints

In the case of our website, significant constraints revolve around the integration and utilization of various APIs from social media platforms. These APIs dictate the methods by which we can access and aggregate photos from the choosen platforms. In our research, we inquired about several social media applications, focusing on their guidelines and limitations imposed in the APIs (such as rate limits, data usage restrictions, authentication requirements, and compliance with platform policies). For example, we found out that Instagram limits the usage of aggregation to 150 photos per hour.

With the information presented above, we can conclude that we will not face any timing or memory limitations, as the APIs restrictions are the main concern of our product.

As for the database, we will use PostgreSQL. This database will store all of our users' data, including their passwords which will be hashed before saving in order to maintain the bare minimum of security. Regarding the security, the users inputs will be treated as normal text in order to avoid code injection.

3.5 User Documentation

Our users can find a README.md file on the previously specified GitHub Repository. The choosen design for our product should be simple enough to make the user experience as intuitive as possible, thus overcomind the need for documentation. However, there will be a page which will inform all the users of the limitations imposed by our website, mainly the number of photos that can be collected in a period of time from another application. If any, the limitations of uploading a photo to these websites will be also documented.

4. External Interface Requirements

4.1 User Interfaces

This chapter describe the logical characteristics of each interface between the software product and the users. Our website aims to use a minimalist style in order to make the user flow/navigation simpler. The choosen design should enhance the user experience. Because the design is so simple, there should be a minimal difference between the Web and Mobile versions. Even the navbar on the left can be permanently visible, as it's width is negligible.

4.1.1. Main Components

There are quite a few software components that require a user interface. They will be presented one by one with a short description and a design concept in the form of a photo. In the later chapters they will be presented in depth if needed.

  1. Home Page

    Starting from the Home Page, the user will be greeted by a small presentation about the website. It will contain the name of the application, its role and a link to this documentation. If the user scrolls down, all of the application features will be visible. The footer of the page will contain relevant data about the website.

  2. Connect Page

    If the user wants to continue navigating the site, he needs to log in. The same page will be used both for signing up and logging in, therefore the user will switch between these two options. These forms will be in a standard one, requiring the minimum information in order to be able to log in and proceed to the site. There will be 3 versions of this form, depending on the screen size. A primitive concept can be found below.

  3. Link a platform

    After logging to our platform, in order to be able to aggregate photos, the user has to connect to at least one social media platform. There will be a page where the user has to select which platform to log into. From there, he will be redirected to the official log in page of the choosen platform.

  4. Aggregation page

    This is the main page the user will work on after successfully logging in with an associated platform. Aggregated photos will be displayed as a mosaic grid. In order to access the comments from a specific post, the user will click the corresponding photo. To select a couple of photos to create a collage, the user has to use the following shortcut on each desired photo: Ctrl+LMB. These features will be presented in the next chapter.

4.1.2. Standards

Our website relies on minimalist buttons, using two types of clickable buttons. The first one is a normal button with relevant text in it. The radius should be 5% of the button width. The second type of buttons are icons - these will be the main components that the user clicks to interact with our webpage.

As for the errors, they will be thrown as a pop-up div with a slightly red background (high transparency).

4.2 Hardware Interfaces

M-PIC is designed to be accessible on any screen device as said before. Responsive design principles are implemented to ensure optimal viewing and interaction, regardless the screen size and resolution.

The website interact with hardware components to receive user input (e.g. mouse clicks, touch events for phones/tablets, swipes, keyboard inputs) and transmit data to and from the server. The photos will be transmited to the server among with their metadata, using Hypertext Transfer Protocol Secure (HTTPS).

4.3 Software Interfaces

M-PIC will not use any already built frameworks, it will rely on its own framework: regacip. However, it will use RESTful APIs to facilitate communication between the M-PIC website and external services, the most obvious one being the supported social media platforms. As for the database, we have choosen to use PostgreSQL - it follows the relational model and stores data in tables with rows and columns, allowing for efficient storage and retrieval of structured data.

This application will be available on any operating system, the only requirement being an operational browser. M-PIC will implement session management techniques to maintain user authentication state.

4.4 Communication Interfaces

Our API requests and responses must be encrypted using HTTPS to protect user data and secure communication.

5. System features

The main features of the application M-PIC are described below. Each feature will be detailed with information about how it will work in the final product. If applicable, there will also be a photo to the choosen design for the interface.

5.1 Aggregation

The design for this component was presented in the previous chapter. However, unlike the log in system, there are still things to be covered. In order to access this page, the user needs to connect to an external social media platform (Instagram or Youtube). After a successful login, the user can select the number of aggregated photos. The default value will be 10 and they will be sorted by the post date, the newest ones coming first. If the user wants to retrieve a specific photo, it will be choosen via an ID.

5.2 View post details

Each photo or video from the grid can be clicked on. When selected, a caroussel style pop-up with cards will show up, bluring the rest of the screen. Under the selected pop-up card, there will be two neighbour cards - slightly rotated and blured. The buttons on these side cards won't work. These cards will contain relevant info about the post (like comments, number of likes, a share button etc). There will also be two arrows (left and right) that will allow the user to iterate through his aggregated media files. This action will move the cards to left or right, simulating a caroussel. This feature will be available on phones too, the only difference being that there won't be an animation, the two neighbour cards being hidden. To exit this cards interface the user can click on any part of the blurred website or on the X button of the current card.

The structure of the card is as follows (from top to bottom): photo, description, number of likes & utility buttons (share, edit), comments (will have an invisible scrollbar if there is too much text).

5.3 Multi-criteria search

The aggregated media will store more than the photo itself - there will also be the post date, number of likes, comments, dimensions etc. Therefore, an user will have the ability to search between his photos/videos using more than one keyword. Also, the results can be filtered based on the platform, a data range etc. The search bar will be visible at all times on the Desktop version as well as on the phone version. However, the top navbar will suffer a loss on the phone version - the current page name won't be displayed anymore to let room for the search bar.

5.4 Apply filters

On the pop-up cards that appear when selecting a photo, next to the share button there will be an edit icon. When clicked, a div will appear with some sliders. These sliders will control different properties of the photo, such as brightness, blur, contrast etc. The user will also have the option to save the edited image in his personal device. When this option is active, the edit icon will have the color slightly changed to let the user know it is pressed. Hopefully, the design is intuitive enough to let the user know to press again when he wants to exit the editor.

5.5 Create collage

As mentioned before, the user has the ability to create collages from his photos. In order to achieve this, the desired photos must be selected using the keyboard shortcut Ctrl+LMB. When a photo is selected, it will be zoomed in, a checkmark will apear on a top corner of the card. The photo can be deselected by following the same shortcut. If the user accidentally clicks on a photo without using the Control key, he can exit from the pop-up modal and continue from where he left.

Also, a button will appear at the bottom of the screen allowing the user to create a collage and save it afterwards.

5.6 Media upload

On the left navbar, there will (eventually) be a button that allows the user to upload a photo to a linked social media account. The user needs to complete a form, uploading the desired photo, writing a description if necessary and other relevant information.

6. Other Nonfunctional Requirements

6.1 Performance Requirements

The M-PIC website must have a loading time as low as possible, this aspect being crucial for providing a seamless user experience. This loading time could be measured using Google PageSpeed Insights or other similar tools. For the average sized photos, our website should be able to load them in under 2 or 3 seconds.

The multi-criteria search function should be able to return search results within a matter of seconds in the worst case, under average server load conditions.

6.2 Safety Requirements

As our site works with media from other associated platforms, there are risks on aggregating their content, such as data breaches, API misuse or platform policy violations. However, these risks should be taken care of by the used APIs (they should prevent unauthorized access to user data, rate limits, etc.).

M-PIC must comply with regulations such as GDPR to ensure protection of user data (secure authentification mechanisms, ecyption of sensitive data). As for the uploaded or aggregated photos, M-PIC is only an intermediary application, therefore the linked social media applications will have control over the distributed photos. M-PIC won't allow the user to edit photos in order to impersonate, allowind the user only to make minor changes to his photos (such as modifying the contrast, hue, blur).

This web application won't have access to sensitive data from the linked social media accounts, as the connection will be made externally.

6.3 Security Requirements

M-PIC will encrypt all sensitive user data, including passwords and personal information, both during transmission and storage. Also, it will use secure communication protocols, such as HTTPS, for all data transmission between clients and server.

M-PIC website must implement measures to prevent code injection vulnerabilities, such as SQL Injection or command injection. Input validation and sanitization must be performed on all user-supplied data to ensure that it does not contain malicious code.

6.4 Business Rules

The final product won't have different types of users, they will respect the same rules on the website. All of them will need to log into our website and later on link a social media platform. There won't be any given restrictions to the registred accounts. Only the developers will have access to the website's database.
